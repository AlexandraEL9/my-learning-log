Combining Database Paradigms

Jargon

Term Definition
Tables Containers for storing data, usually represented in a row/column format.
Row An individual record of data. This could contain your name, address, and phone number, for example.
Column An individual field within the data, such as name, phone number, email address, etc.
Schema The blueprint of how the data is to be stored. The metadata of the tables as well as its definition in a diagram.
Relational Database A type of database that organizes data in multiple tables that can be linked - or related - based on data common to each.
Non-Relational Database A type of database that does not store its data in rows and columns and does not manage links between the data.
DBMS / RDBMS Database Management System or Relational Database Management Systems. The software like MongoDB or PostgreSQL that allows you to manage your database.

Structured data is highly organized and formatted in a way that is easily searchable in databases. Examples include relational databases like SQL, where data is stored in tables with rows and columns. Non-structured data, on the other hand, lacks a predefined format or organization, making it more difficult to collect, process, and analyze. Examples include text files, emails, videos, and social media posts.

PostgreSQL vs MongoDB

PostgreSQL MongoDB
  • Define the schema in advance
  • Harder to change later
  • Relationships keep data structured
  • Suitable for data like user details, sales data, invoices, and addresses
  • Schema can be defined on the fly
  • Every record can have its own schema
  • Manually manage relationships
  • Suitable for data like tweets, social media posts, or LMS course content

PostgreSQL is a powerful, open-source object-relational database system that uses and extends the SQL language. It is known for its reliability, robustness, and performance. MongoDB is a NoSQL database that stores data in flexible, JSON-like documents. It is designed to scale out and handle large amounts of data across distributed systems.

Relational vs Non-Relational- Which is Best?

The answer is... it depends

  • 1. Use a relational database like PostgreSQL when your data is highly structured.
  • 2. Use a non-relational database like MongoDB when you need to store document-type information.
  • 3. You can structure MongoDB to act relationally, but it is not a relational database system.

Combining Database Management Systems

In some cases, you may need to use both relational and non-relational databases in your application. This is known as a polyglot persistence strategy. For example, you might use a relational database to store user account information and a non-relational database to store user-generated content like social media posts or comments.
